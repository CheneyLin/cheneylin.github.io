2. 背景与痛点

付邮购活动允许用户以 0 元商品价参加，但需支付运费。目前付邮购仅支持单一价格 SKU，无法支持升款服务。

随着营销活动对用户体验和转化的要求提升，需要在付邮购活动中支持“基础 SKU + 升款 SKU”形式，让用户在保留付邮购免费商品体验的同时，通过选择升款服务获取更高价值商品 / 配件。

现状问题

  1. 付邮购仅支持单一价格 SKU，SKU 价格不能差异化；
  2. 无法将高于基础价格的部分标记为“升款服务”；
  3. 费用展示仅为“运费”，无法清晰区分基础款与升款服务。

4. 用户与场景

核心用户

  • 运营人员：负责在后台营销管理中配置付邮购商品、设置基础售价和升款 SKU。
  • 购买用户：在小芒定制产品页选择付邮购商品，并在商品详情页查看升款服务价格及费用构成。
  • 订单用户：在订单详情页查看最终费用构成，确认升款服务金额与运费。

场景说明

  • 运营人员在后台操作：为付邮购活动关联 SPU，填写基础价格，新增升款 SKU 作为可选项。
  • 用户在小芒定制产品页参与活动：默认展示付邮购基础 SKU，并可从升款项列表中切换选择一个升款服务。
  • 用户在订单详情页查看：订单费用展示“升款服务 + 运费 + 总价”，明确区分费用来源。

6. 详细功能需求

6.1 运营后台

6.1.1 运营后台 - 付邮购管理模块

配置逻辑
  • 以 SPU 为粒度创建付邮购活动。
  • 运营配置 SPU 底价（基础价格）。
  • 运营可新增“升款 SKU”，每个升款 SKU 定价高于底价，差价部分视为升款服务金额。
  • 升款 SKU 类型为“升款服务大项”，示例子项：挂钩 4.00 元。
  • 仅支持用户选择一个升款 SKU，不能同时叠加多个升款服务。
  • 若底价未填写，则进入兼容旧逻辑：所有 SKU 价格按单一价格处理，价格差异不作为升款服务，全部计入运费显示。

6.2 客户端

6.2.1 商品详情页

展示需求
  • 商品详情页需展示基础 SKU 与升款 SKU 的价格差异。
  • 升款 SKU 价格需明确标识为“升款服务”，并展示升款服务金额。
  • 费用展示需区分“基础商品价格（0 元）”、“升款服务金额”和“运费”，让用户清晰了解费用构成。

6.2.2 小芒定制产品页

交互逻辑
  • 用户进入小芒定制产品页后，默认展示基础 SKU 与升款服务项列表。
  • 用户可通过切换 SKU 实现升款服务：基础款 SKU 显示“运费、总价”，升款 SKU 显示“升款服务、运费、总价”。
  • 展示方式强调“基础商品免费 + 付邮领”的体验，同时通过升款服务项提供升级入口。
  • 费用明细中，基础款仍集中展示在运费项，升款则单独展示升款服务金额。
  • 订单详情页需要同步展示：升款服务金额、运费、总价。

6.2.3 订单详情页

显示需求
  • 订单详情页需同步展示：升款服务金额、运费、总价。

7. 业务流程图

系统流程说明：运营配置 SPU 底价与升款 SKU，用户在小芒定制产品页切换 SKU，最终订单展示升款服务与运费构成。
